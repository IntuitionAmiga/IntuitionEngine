# 6502 JIT Compiler

Technical reference for the MOS 6502 Just-In-Time compiler. Covers the context struct, block scanner, x86-64 backend, dispatcher, cycle accounting, and bail/invalidation semantics.

For the matching **fast interpreter** that handles non-JIT runs (debug mode, decimal-mode ADC/SBC, undocumented opcodes that bail from a JIT block), see `cpu_six5go2_fast.go`. The fast interpreter shadows hot CPU state in loop-local variables, reuses the same `directPageBitmap` for translation-safe memory access, and inlines the full documented opcode set. `CPU_6502.Execute()` routes non-debug runs to `ExecuteFast()` when `fastAdapter != nil` and otherwise falls through to `executeLegacy()` (the original generic opcode-table interpreter).

---

## Overview

The 6502 JIT compiler translates blocks of 6502 machine code into native x86-64 (ARM64 planned) instructions at runtime, executing them directly on the host CPU. This bypasses the Go interpreter's per-instruction dispatch overhead (opcode table lookup, interface method calls) and yields 7-13x speedup on compute-bound workloads — measured against the legacy generic interpreter. Against the fast interpreter (`ExecuteFast()`), the JIT delivers roughly 2.3x (Call) to 8.4x (ALU) on the comparison benchmark suite; see DEVELOPERS.md §9 for the current reference table.

**Supported platforms:** x86-64/Linux and x86-64/Windows (ARM64 backend planned)

**Activation:** Set `CPU6502Runner.JITEnabled = true` or `cpu.jitEnabled = true`.

**Coverage:** All documented 6502 opcodes except BRK and RTI (~150 of 256). Undocumented opcodes and decimal-mode ADC/SBC bail to the interpreter.

**JIT call path:** The Go side of the JIT trampoline (`jit_call.go`) dispatches through `runtime.asmcgocall` — the raw g0 stack-switch primitive — rather than `runtime.cgocall`. `asmcgocall` has no `iscgo` guard, so the JIT works identically in `CGO_ENABLED=1` builds (the normal dev build) and `CGO_ENABLED=0` builds (the fully static portable benchmark binary produced by `build_6502_benchmarks.sh`). The asm trampolines in `jit_call_{amd64,arm64}.s` were already written to the `asmcgocall` contract, so only the Go dispatch side needed changing.

## JIT6502Context

Bridge between Go and native code. Passed as the sole argument to every compiled block (in RDI on x86-64).

```
Offset  Type      Field               Description
0       uintptr   MemPtr              &adapter.memDirect[0]
8       uintptr   IOBitmapPtr         &adapter.ioPageBitmap[0] ([]bool)
16      uintptr   CpuPtr              &cpu (CPU_6502 struct pointer)
24      uintptr   CodePageBitmapPtr   &codePageBitmap[0] (256 bytes)
32      uint64    RetCycles           Accumulated cycles (see Bail Invariant for per-exit semantics)
40      uint32    NeedBail            Set to 1 on bail: RetPC is current instr PC (re-execute)
44      uint32    NeedInval           Set to 1 on self-mod: RetPC is next instr PC (store done)
48      uint32    RetPC               Normal exit: next PC after block.
                                      NeedBail: current instruction's PC (to be re-executed).
                                      NeedInval: next instruction's PC (store already completed).
52      uint32    RetCount            Normal exit: total instructions retired.
                                      NeedBail: instructions retired BEFORE the bailing instruction.
                                      NeedInval: instructions retired INCLUDING the completed store.
56      uint32    FastPathLimit       0x2000 (Tier 1 memory boundary)
60      uint32    (padding)           Alignment
```

Unlike IE64, the 6502 JIT uses separate `RetPC`/`RetCount`/`RetCycles` fields (not a packed `regs[0]` return channel) because the 6502 has no zero register to repurpose.

## Block Scanner

`jit6502ScanBlock()` walks variable-length 6502 instructions (1-3 bytes each) using the `jit6502InstrLengths[256]` table. Stops at:

- **Block terminators** (included in block): JMP ($4C, $6C), JSR ($20), RTS ($60), RTI ($40), BRK ($00), KIL/JAM opcodes
- **Uncompilable opcodes** (block ends BEFORE them): all undocumented/illegal opcodes. The uncompilable instruction starts the next block, where `needsFallback()` routes it through the interpreter.
- **Maximum block size:** 128 instructions

Conditional branches (BCC/BCS/BEQ/BNE/BMI/BPL/BVC/BVS) are NOT terminators. They are compiled as internal conditional jumps within the block.

### Lookup Tables

| Table | Type | Purpose |
|-------|------|---------|
| `jit6502InstrLengths[256]` | `byte` | Instruction length (1, 2, or 3) from opcode |
| `jit6502BaseCycles[256]` | `byte` | Base cycle cost per opcode (excluding variable penalties) |
| `jit6502IsCompilable[256]` | `bool` | True for Tier 1 opcodes only |

**Branch cycle note:** Branch opcodes ($10,$30,$50,$70,$90,$B0,$D0,$F0) have base cost 0 in `jit6502BaseCycles`. This matches the interpreter's `branch()` function, which adds no base cycles — only +1 for taken and +1 for page-cross.

## x86-64 Backend

### Register Mapping

```
x86-64   6502    Purpose                    Persistence
RBX      A       Accumulator (low byte BL)  callee-saved
RBP      X       X index (low byte BPL)     callee-saved
R12      Y       Y index (low byte R12B)    callee-saved
R13      SP      Stack pointer (R13B)       callee-saved
R14      PC      Program counter (R14W)     callee-saved
R15      SR      Status register (R15B)     callee-saved
RSI      --      memDirect base pointer     dedicated
RDI      --      JIT6502Context* (entry)    saved to stack
R8       --      ioPageBitmap pointer       dedicated
R9       --      FastPathLimit (0x2000)     dedicated
RAX      --      Scratch                    caller-saved
RCX      --      Scratch / shift count      caller-saved
RDX      --      Scratch                    caller-saved
R10      --      Scratch                    caller-saved
R11      --      Scratch                    caller-saved
```

All 6 CPU registers (A, X, Y, SP, PC, SR) are permanently mapped to callee-saved host registers. No register spilling is ever needed.

### Stack Frame Layout

```
RSP+0   = saved JIT6502Context pointer
RSP+8   = saved CpuPtr (from context)
RSP+16  = cycle accumulator (uint32)
RSP+20  = backward branch budget counter (uint32)
RSP+24  = CodePageBitmapPtr (from context)
```

6 callee-saved pushes (48 bytes) + SUB RSP,40 = 88 + 8 (ret addr) = 96 bytes = 16-byte aligned.

### CPU_6502 Struct Offsets

The prologue/epilogue load/store registers directly from the CPU struct via CpuPtr:

```
Offset  Field   Size
0       PC      uint16
2       SP      byte
3       A       byte
4       X       byte
5       Y       byte
6       SR      byte
```

Verified at compile time with `unsafe.Offsetof` in `TestJIT6502_CPUStructFieldOffsets`.

### Byte Register Encoding

x86-64 byte operations on registers 4-7 (SPL, BPL, SIL, DIL) require a REX prefix to access the low byte instead of AH/CH/DH/BH. The emitter uses `emitREXForByte()` which always emits REX when the register index is 4-7 or 8-15.

## Memory Access

### Fast Path

The 6502 uses a `Bus6502Adapter` that translates 16-bit 6502 addresses to 32-bit system bus addresses, with complex I/O routing, bank windows, and VRAM mapping.

**Tier 1 fast path:** `addr < 0x2000` AND `!ioPageBitmap[addr >> 8]` → direct `memDirect[addr]`

This covers zero page ($00-$FF), stack ($100-$1FF), and low RAM ($200-$1FFF) — the most performance-critical access patterns.

**Everything else bails to the interpreter:** I/O pages, bank windows ($2000-$7FFF), VRAM ($8000-$BFFF), I/O devices ($D000-$DFFF).

**Stack operations** (PHA/PLA/PHP/PLP/JSR/RTS) access $0100-$01FF which is always within the fast path, so they skip the range check entirely.

### Self-Modification Detection

A 256-byte code page bitmap (one byte per 6502 page) tracks which pages contain cached JIT blocks. After every STORE, the emitter checks `codePageBitmap[addr >> 8]`. If the page has cached code, `NeedInval` is set and the dispatcher invalidates the entire cache.

Stack pushes and zero-page data writes typically hit non-code pages (bitmap=0), so the compare falls through with minimal overhead (a single byte compare + untaken branch per store).

## Bail Invariant

There are two distinct bail paths:

### NeedBail (re-execute bail)

Used for I/O memory access, decimal-mode ADC/SBC, and any condition requiring full interpreter semantics.

1. `RetPC` = current instruction's PC (to be re-executed)
2. `RetCount` = instructions retired BEFORE this instruction
3. `RetCycles` = cycles accumulated BEFORE this instruction
4. Dispatcher calls `interpret6502One()` which re-executes the instruction

### NeedInval (self-modification bail)

Used after a STORE hits a code page. The store has already completed.

1. `RetPC` = NEXT instruction's PC (store already done)
2. `RetCount` = instructions retired INCLUDING the store
3. `RetCycles` = cycles accumulated INCLUDING the store's cost
4. Dispatcher invalidates cache, does NOT re-execute any instruction

## Cycle Accounting

The JIT produces exactly the same `cpu.Cycles` value as the interpreter for any instruction sequence. The interpreter is the source of truth.

**Fixed costs:** Each instruction's base cycle count (from `jit6502BaseCycles`) is accumulated in a compile-time `pendingCycles` counter and flushed to the runtime accumulator `[RSP+16]` periodically. Consecutive fixed-cost instructions are batched into a single ADD.

**Variable costs** (emitted as runtime conditional increments):
- **Absolute,X / Absolute,Y** (loads only): +1 if page crossed
- **(Indirect),Y** (loads only): +1 if page crossed
- **Conditional branches**: +1 if taken, +1 more if page crossed (both are compile-time constants since branch targets are known at compile time)

**Cycle flush points:** Pending cycles are flushed to the runtime accumulator:
- Before any branch instruction (critical for loops)
- At any instruction that is an internal branch target
- Before any bail point
- At block exit (normal epilogue)

**Dispatcher handling:** After each block, `cpu.Cycles += ctx.RetCycles`. On NeedBail, `interpret6502One()` mutates `cpu.Cycles` directly via the opcode handler.

---

## Conditional Branches

Branches test a specific SR flag bit and conditionally jump to a target:

| 6502 | Flag | Taken when |
|------|------|------------|
| BCC | C (0x01) | clear |
| BCS | C (0x01) | set |
| BEQ | Z (0x02) | set |
| BNE | Z (0x02) | clear |
| BMI | N (0x80) | set |
| BPL | N (0x80) | clear |
| BVS | V (0x40) | set |
| BVC | V (0x40) | clear |

**Internal branches** (target within block):
- Forward: emit JMP with forward fixup, patched after all instructions are emitted
- Backward: emit JMP to known `instrOffsets[targetIdx]`, with budget check (4095 iterations)

**External branches** (target outside block): emit deferred epilogue with `RetPC = targetPC`.

## Dispatcher Loop

```
ExecuteJIT6502():
  Pre-loop: ensureOpcodeTableReady, SealMappings, fastAdapter check, initJIT6502

  for running:
    1. Check resetting signal, RDY line (every block boundary)
    2. Check NMI/IRQ interrupts (before fetch)
    3. Cache lookup → hit or scan+compile+cache
    4. If needsFallback → interpret6502One(), continue
    5. callNative(block.execAddr, ctx)
    6. cpu.Cycles += ctx.RetCycles
    7. Handle NeedInval → cache.Invalidate + execMem.Reset + clear bitmap
    8. Handle NeedBail → interpret6502One()
    9. Performance reporting
```

**Debug mode:** When `cpu.Debug` is true, `jit6502Execute()` falls back to the interpreter entirely. Debug breakpoints require per-instruction checks that are incompatible with block-level JIT execution.

## Fallback Rules

| Condition | Mechanism |
|-----------|-----------|
| BRK, RTI as first instruction | `needsFallback()`, dispatcher calls `interpret6502One()` |
| KIL/JAM opcodes | `needsFallback()`, interpreter sets `running=false` |
| Undocumented opcodes | Scanner stops before them; `needsFallback()` on next block |
| Decimal-mode ADC/SBC | Runtime D-flag check → NeedBail |
| I/O / bank / VRAM address | Fast-path check fails → NeedBail |
| Self-modifying code | Post-STORE bitmap check → NeedInval |
| ExecMem exhausted | `compileBlock6502` returns error → `interpret6502One()` |
| Debug mode | Entire execution uses interpreter |

## Sample Benchmark Results

Results from a single run on one machine. Performance varies with CPU microarchitecture, Go version, and system load. Re-run with `-benchtime 3s` for your own baseline.

**Intel Core i5-8365U @ 1.60GHz, Go 1.24, `-benchtime 2s`, 2025-03-28:**

| Workload | Instrs/run | Interpreter ns/op | JIT ns/op | Speedup | Interp MIPS | JIT MIPS |
|----------|-----------|-------------------|-----------|---------|-------------|----------|
| ALU | 2,306 | 23,756 | 2,162 | **11.0x** | 97 | 1,067 |
| Memory | 1,025 | 12,142 | 912 | **13.3x** | 84 | 1,124 |
| Call | 1,281 | 16,705 | 55,330 | 0.30x | 77 | 23 |
| Branch | 1,537 | 15,183 | 1,445 | **10.5x** | 101 | 1,064 |

**Call workload regression (0.30x):** Expected. JSR/RTS are block terminators, so every call/return exits the JIT block and returns to the Go dispatcher. ## Performance Guardrails

### Built In
- All 6 CPU registers permanently mapped to callee-saved host registers (no spilling)
- Cycle batching: consecutive fixed-cost instructions emit a single ADD
- Branch target cycle flush: pending cycles flushed only at branch targets
- Stack operations skip fast-path range check ($0100-$01FF always valid)
- Code page bitmap: single byte compare per STORE, cheap fall-through for non-code pages
- Deferred bail/inval epilogues: cold code at end of block, fast path falls through

### Deferred
- Expand memory fast path beyond $2000 (check ioPageBitmap for all pages)
- Popular undocumented opcodes (LAX, SAX, DCP, ISC, SLO, SRE, RLA, RRA)
- ARM64 backend
- Decimal-mode ADC/SBC compilation

### Not Planned
- Dynamic register allocation (unnecessary — only 6 registers)
- Instruction scheduling (6502 instructions are too simple to benefit)
- Block linking (JSR/RTS chaining without returning to dispatcher)

## Host W^X

The 6502 JIT shares the `jit_mmap.go` dual-mapped executable memory
with every other JIT backend. Emit and patch operations run through
the writable view (`PROT_READ|PROT_WRITE`); dispatch runs through the
execution view (`PROT_READ|PROT_EXEC`). At no point does either view
hold both write and execute permission. See
[`IE64_JIT.md`](IE64_JIT.md) for the full model and test contract.
