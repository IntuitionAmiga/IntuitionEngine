# IE32-to-IE64 Assembly Converter

Source-to-source transpiler for the supported IE32 assembly syntax accepted by
`ie32asm`. When conversion completes with zero `; ERROR:` markers, transformed
top-level lines use IE64 syntax intended for `ie64asm`; `include` paths are
emitted as `include` directives and their file contents are not inspected by the
converter.

## Quick Start

```bash
# Build
go build -o bin/ie32to64 ./cmd/ie32to64/

# Convert a file
bin/ie32to64 sdk/examples/asm/rotozoomer.asm
# produces sdk/examples/asm/rotozoomer_ie64.asm

# Convert with explicit output path
bin/ie32to64 -o output.asm sdk/examples/asm/rotozoomer.asm

# Upgrade to 64-bit operations
bin/ie32to64 -size .q sdk/examples/asm/program.asm

# Print conversion statistics
bin/ie32to64 -stats sdk/examples/asm/rotozoomer.asm
```

## CLI Reference

```
Usage: ie32to64 [options] input.asm

Options:
  -o output.asm    Output file (default: input_ie64.asm)
  -size .l|.q      Default size suffix (default: .l, use .q for 64-bit)
  -no-header       Omit "Converted from IE32" header comment
  -no-div-guard    Disable DIV/MOD zero guards
  -stats           Print conversion statistics
  -h               Help
```

## Register Mapping

IE32's 16 named registers map to IE64 `r1`-`r16`. IE64 `r0` is the hardwired
zero register. The converter may emit `r17` and `r18` as scratch registers, so
converted code must treat them as clobbered at each expansion site. The current
converter does not emit `r19`-`r30`. IE64 `r31` is the architectural stack
pointer used by generated `push` and `pop` instructions.

| IE32 | IE64 | IE32 | IE64 |
|------|------|------|------|
| A    | r1   | F    | r9   |
| X    | r2   | G    | r10  |
| Y    | r3   | H    | r11  |
| Z    | r4   | S    | r12  |
| B    | r5   | T    | r13  |
| C    | r6   | U    | r14  |
| D    | r7   | V    | r15  |
| E    | r8   | W    | r16  |

**Scratch registers**: `r17` is used for direct-address loads/stores,
memory-operand ALU lowering, and DIV/MOD guard materialisation. `r18` is used
for memory `INC`/`DEC` read-modify-write lowering.

## Size Suffix

The `-size` option selects the suffix used on generated IE64 integer
`move`, `load`, `store`, ALU, `not`, `INC`/`DEC`, and DIV/MOD guard
materialisation instructions. It accepts only `.l` and `.q`; the CLI rejects any
other value before conversion. It does not change generated branches, `push`,
`pop`, `wait`, labels, or data directives.

## Instruction Reference

### Zero-Operand Instructions

| IE32 | IE64 |
|------|------|
| `NOP` | `nop` |
| `HALT` | `halt` |
| `RTS` | `rts` |
| `SEI` | `sei` |
| `CLI` | `cli` |
| `RTI` | `rti` |

### Load Instructions

Register-specific loads cover all 16 mapped IE32 registers: `LDA`, `LDX`,
`LDY`, `LDZ`, `LDB`, `LDC`, `LDD`, `LDE`, `LDF`, `LDG`, `LDH`, `LDS`, `LDT`,
`LDU`, `LDV`, and `LDW`. Generic `LOAD` takes the destination register as its
first operand.

| IE32 | IE64 | Notes |
|------|------|-------|
| `LDA #42` | `move.l r1, #42` | Immediate |
| `LDA @0x5000` | `la r17, 0x5000` + `load.l r1, (r17)` | Memory dereference |
| `LDA 0x5000` | `move.l r1, #0x5000` | Bare number = immediate |
| `LDA MY_CONST` | `move.l r1, #MY_CONST` | Bare equate = immediate |
| `LDA [B]` | `load.l r1, (r5)` | Register-indirect |
| `LDA [B+16]` | `load.l r1, 16(r5)` | Register+offset; current `ie32asm` requires IE32 register-indirect offsets to be multiples of 16 |
| `LDA [B-16]` | `load.l r1, -16(r5)` | Negative register offset |
| `LDA X` | `move.l r1, r2` | Register-to-register |
| `LOAD A, #42` | `move.l r1, #42` | Generic load, immediate |
| `LOAD A, X` | `move.l r1, r2` | Generic load, register |
| `LOAD A, @0x5000` | `la r17, 0x5000` + `load.l r1, (r17)` | Generic load, dereference |

### Store Instructions

Register-specific stores cover all 16 mapped IE32 registers: `STA`, `STX`,
`STY`, `STZ`, `STB`, `STC`, `STD`, `STE`, `STF`, `STG`, `STH`, `STS`, `STT`,
`STU`, `STV`, and `STW`. Generic `STORE` takes the source register as its first
operand.

| IE32 | IE64 | Notes |
|------|------|-------|
| `STA @0x5000` | `la r17, 0x5000` + `store.l r1, (r17)` | Memory write via @ |
| `STA 0x5000` | `la r17, 0x5000` + `store.l r1, (r17)` | Bare addr = memory write |
| `STA [B]` | `store.l r1, (r5)` | Register-indirect |
| `STORE A, @0x5000` | `la r17, 0x5000` + `store.l r1, (r17)` | Generic store |
| `STORE A, [B]` | `store.l r1, (r5)` | Generic store, indirect |
| `STORE A, X` | **ERROR** | The converter refuses register destinations for STORE |
| `STORE A, #5` | **ERROR** | Immediate STORE is invalid |

**Key semantic rule**: In IE32 assembly, `LOAD A, 0x5000` loads the value
`0x5000` into `A` because a bare resolved expression is encoded as an immediate
operand. The current IE32 CPU `STORE` path writes generic and register-specific
stores to the operand address for the bare operand form, and the converter
therefore lowers `STORE A, 0x5000` as a memory write to address `0x5000`. Use
`@` on `LOAD` when the source value must be read from memory.

### ALU Instructions (2-operand to 3-operand)

IE32's `OP reg, operand` becomes IE64's `op.l reg, reg, operand` with the
default `-size .l` setting:

| IE32 | IE64 |
|------|------|
| `ADD A, #10` | `add.l r1, r1, #10` |
| `ADD A, X` | `add.l r1, r1, r2` |
| `SUB A, #5` | `sub.l r1, r1, #5` |
| `MUL A, #3` | `mulu.l r1, r1, #3` |
| `DIV A, #2` | `divu.l r1, r1, #2` |
| `MOD A, #7` | `mod.l r1, r1, #7` |
| `AND A, #0xFF` | `and.l r1, r1, #0xFF` |
| `OR A, #0x80` | `or.l r1, r1, #0x80` |
| `XOR A, #0xFF` | `eor.l r1, r1, #0xFF` |
| `SHL A, #4` | `lsl.l r1, r1, #4` |
| `SHR A, #4` | `lsr.l r1, r1, #4` |
| `NOT A` | `not.l r1, r1` |

ALU with memory operands expand to multi-line sequences:

```asm
; IE32: ADD A, @addr
la r17, addr
load.l r17, (r17)
add.l r1, r1, r17

; IE32: ADD A, [B]
load.l r17, (r5)
add.l r1, r1, r17

; IE32: ADD A, [B+16]
load.l r17, 16(r5)
add.l r1, r1, r17
```

### Divide/Modulo by Zero

IE32 stops execution when `DIV` or `MOD` sees a zero divisor. The IE32 CPU path
sets `running` to false for this case. IE64 divide/modulo instructions do not
provide the same halt-on-zero behaviour, so the converter guards every divisor
that is not a parsed non-zero integer literal.

Static literal zero emits `halt` directly:

```asm
; IE32: DIV A, #0
halt
```

Register, direct memory, and register-indirect divisors emit a zero branch before the IE64 operation:

```asm
; IE32: DIV A, X
beqz r2, __ie32to64_div_zero_0
divu.l r1, r1, r2
bra __ie32to64_div_skip_0
__ie32to64_div_zero_0:
halt
__ie32to64_div_skip_0:

; IE32: DIV A, @addr
la r17, addr
load.l r17, (r17)
beqz r17, __ie32to64_div_zero_1
divu.l r1, r1, r17
bra __ie32to64_div_skip_1
__ie32to64_div_zero_1:
halt
__ie32to64_div_skip_1:

; IE32: MOD A, [B-16]
load.l r17, -16(r5)
beqz r17, __ie32to64_div_zero_2
mod.l r1, r1, r17
bra __ie32to64_div_skip_2
__ie32to64_div_zero_2:
halt
__ie32to64_div_skip_2:
```

Symbolic immediates and bare expressions are materialised into `r17` before the
guard because the converter does not evaluate expressions:

```asm
; IE32: DIV A, FOO
move.l r17, #FOO
beqz r17, __ie32to64_div_zero_3
divu.l r1, r1, r17
bra __ie32to64_div_skip_3
__ie32to64_div_zero_3:
halt
__ie32to64_div_skip_3:
```

Use `-no-div-guard` only when you deliberately want unguarded IE64 `divu`/`mod`
output and have proved that zero divisors cannot occur.

### INC/DEC

| IE32 | IE64 |
|------|------|
| `INC A` | `add.l r1, r1, #1` |
| `DEC A` | `sub.l r1, r1, #1` |

Direct memory `INC`/`DEC` expand to a four-line read-modify-write sequence:

```asm
; IE32: INC @0x5000
la r17, 0x5000
load.l r18, (r17)
add.l r18, r18, #1
store.l r18, (r17)
```

Register-indirect memory `INC`/`DEC` do not need an address materialisation line:

```asm
; IE32: INC [B]
load.l r18, (r5)
add.l r18, r18, #1
store.l r18, (r5)
```

| IE32 | IE64 | Notes |
|------|------|-------|
| `JMP label` | `bra label` | Unconditional |
| `JSR label` | `jsr label` | Subroutine call |
| `JNZ A, label` | `bnez r1, label` | Branch if not zero |
| `JZ A, label` | `beqz r1, label` | Branch if zero |
| `JGT A, label` | `bgtz r1, label` | Branch if greater than zero |
| `JGE A, label` | `bgez r1, label` | Branch if greater or equal zero |
| `JLT A, label` | `bltz r1, label` | Branch if less than zero |
| `JLE A, label` | `blez r1, label` | Branch if less or equal zero |

### Stack Instructions

| IE32 | IE64 |
|------|------|
| `PUSH A` | `push r1` |
| `POP A` | `pop r1` |

### WAIT

| IE32 | IE64 | Notes |
|------|------|-------|
| `WAIT #100` | `wait #100` | Immediate passthrough |
| `WAIT #DELAY_US` | `wait #DELAY_US` | Equate with # |
| `WAIT DELAY_US` | `wait #DELAY_US` | Bare equate: prepend `#` |
| `WAIT 0x1000` | `wait #0x1000` | Bare hex: prepend `#` |
| `WAIT A` | **ERROR** | IE64 `wait` is immediate-only |
| `WAIT @0x5000` | **ERROR** | IE64 `wait` is immediate-only |
| `WAIT [B]` | **ERROR** | IE64 `wait` is immediate-only |

The IE32 assembler can encode non-immediate `WAIT` operands because the normal
IE32 execution path resolves the operand before sleeping. The normal IE32 and
IE64 wait paths sleep in microseconds; single-step paths skip the sleep. The
converter rejects non-immediate forms because IE64 `wait` accepts an immediate
operand only. Convert register or memory waits to an explicit busy-wait loop
before running `ie32to64`.

## Directive Reference

| IE32 | IE64 |
|------|------|
| `.org 0x1000` | `org 0x1000` |
| `.equ NAME 0x1234` | `NAME equ 0x1234` |
| `.equ NAME BASE+8` | `NAME equ BASE+8` |
| `.word v1, v2` | `dc.l v1, v2` |
| `.byte v1, v2` | `dc.b v1, v2` |
| `.space 256` | `ds.b 256` |
| `.ascii "hello"` | `dc.b "hello"` |
| `.asciz "hello"` | `dc.bz "hello"` |
| `.incbin "file"` | `incbin "file"` |
| `.incbin "f", off, len` | `incbin "f", off, len` |
| `.include "file"` | `include "file"` (rewrites `"ie32.inc"` to `"ie64.inc"` only for that exact argument) |

Note: `.word` in IE32 emits 32-bit values (unlike conventional assemblers), so it maps to `dc.l` (not `dc.w`).

`include` and `incbin` paths are emitted verbatim except for the exact
`"ie32.inc"` rewrite above. If `-o` writes the converted file to a different
directory than the source, relative paths may no longer resolve. Prefer writing
output next to the source or pre-rewrite paths.

## Operand Semantics

Understanding IE32's operand resolution is critical for correct conversion:

| Prefix | Type | Meaning |
|--------|------|---------|
| `#` | Immediate | Literal value |
| `@` | Direct | Memory dereference at address |
| `[R]` | Register-indirect | Memory at address in register |
| `[R+N]` | Register+offset | Memory at register + N |
| `[R-N]` | Register-offset | Memory at register - N |
| *(none)* | Bare | Context-dependent (see below) |

**Bare operands** (no prefix):
- In `LOAD`: bare values are **immediate** - `LOAD A, 0x5000` puts `0x5000` into `A`
- In `STORE`: bare values are **memory addresses** - `STORE A, 0x5000` writes `A` to address `0x5000`
- In ALU ops: bare values are **immediate** - `ADD A, MY_CONST` adds the constant value
- Bare register names are recognised as registers where that operand position
  permits a register. For example, `LOAD A, X` copies `X` into `A`; `STORE A, X`
  is still rejected because the converter does not permit register destinations
  for stores.

The converter itself passes register-indirect offsets through as IE64
displacements. Valid source intended for `ie32asm` must still obey the IE32
assembler restriction that `[R+N]` and `[R-N]` offsets are multiples of 16.

## Known Limitations

1. **No `.include` expansion** - the converter processes files line by line
   without expanding includes. It rewrites the exact include argument
   `"ie32.inc"` to `"ie64.inc"`. Other include files must be converted
   separately if they contain IE32 syntax.

2. **No semantic analysis** - conversion is line based and has no register
   liveness tracking. Scratch registers `r17` and `r18` are reloaded before each
   generated use. If hand-written IE64 code is interleaved with converted code,
   verify that it does not rely on `r17` or `r18` surviving across converted
   instructions.

3. **Expressions pass through unchanged** - `RING_BASE + 1` is emitted as-is.
   This is intentional for expression forms accepted by `ie64asm`.

4. **Unknown directives, unknown mnemonics, and unsupported operands produce
   conversion errors** - the output contains `; ERROR:` followed by the original
   source line as a comment, and the CLI exits non-zero after writing the output
   file. Fix the source to valid supported IE32 syntax before assembling the
   converted file.

5. **One comma per instruction operand split** - instruction operands are split
   at the first comma outside square brackets. Do not rely on commas inside
   source expressions.

## Reserved Identifiers

Labels starting with `__ie32to64_div_` are reserved for generated DIV/MOD guard scaffolding. User code must not define labels with this prefix.

## Post-Conversion Checklist

After running the converter:

1. **Review multi-line expansions** - search for `r17` and `r18` to find expansion points; verify scratch register usage does not conflict with surrounding hand-written code
2. **Check for `; ERROR:`** - search the output for error markers; fix source and reconvert
3. **Test assembly** - run `ie64asm` on the output to verify it assembles cleanly
4. **Consider optimisation** - IE64's 3-operand format and `r0` zero register enable optimisations the converter does not perform, for example `move.l r1, r0` instead of `move.l r1, #0`

## Example: Before and After

### IE32 (rotozoomer excerpt)

```asm
.include "ie32.inc"
.equ TEXTURE_BASE 0x600000
.org 0x1000

start:
    LDA #1
    STA @VIDEO_CTRL
    LDA #0
    STA @VIDEO_MODE
    JSR load_texture
    LDA #0
    STA @VAR_ANGLE_ACC
    STA @VAR_SCALE_ACC
    JMP main_loop

wait_vsync:
wait_end:
    LDA @VIDEO_STATUS
    AND A, #STATUS_VBLANK
    JNZ A, wait_end
wait_start:
    LDA @VIDEO_STATUS
    AND A, #STATUS_VBLANK
    JZ A, wait_start
    RTS
```

### IE64 (converted output)

```asm
; Converted from IE32 by ie32to64

include "ie64.inc"
TEXTURE_BASE equ 0x600000
org 0x1000

start:
    move.l r1, #1
    la r17, VIDEO_CTRL
    store.l r1, (r17)
    move.l r1, #0
    la r17, VIDEO_MODE
    store.l r1, (r17)
    jsr load_texture
    move.l r1, #0
    la r17, VAR_ANGLE_ACC
    store.l r1, (r17)
    la r17, VAR_SCALE_ACC
    store.l r1, (r17)
    bra main_loop

wait_vsync:
wait_end:
    la r17, VIDEO_STATUS
    load.l r1, (r17)
    and.l r1, r1, #STATUS_VBLANK
    bnez r1, wait_end
wait_start:
    la r17, VIDEO_STATUS
    load.l r1, (r17)
    and.l r1, r1, #STATUS_VBLANK
    beqz r1, wait_start
    rts
```
