# M15.6 Plan: Pre-M16 Kernel Hardening

## Summary

M15.6 is the hardening milestone that sits between M15.5 substrate work
and M16 "Protected Module Subsystem". The moment M16 lands, every library
becomes its own protected server task driven by an exec-owned registry.
The kernel surfaces that touch module identity, lifecycle state,
cross-task ports, shared memory, and host-backed ELF loading all grow at
once. M16 explicitly depends on M15.4/M15.5 hardening as a prerequisite;
M15.6 widens that base so the new M16 surface sits on a kernel that is
already secure against the SMEP/SMAP, W^X, quota, and cleanup classes of
mistake M16 would otherwise inherit.

Full TDD is mandatory for this milestone. No item ships without failing
tests committed before the implementation commit, regressions green on
exit, and the milestone-output docs updated alongside the code/tests
they describe. M15.6 matches the "docs as milestone output" discipline
locked in by M16.

The end state is:

- host-side JIT memory is never simultaneously writable and executable
- IE64 has real SMEP/SMAP-equivalent guards (SKEF, SKAC) plus an
  explicit supervisor-user-access latch (SUA) gated through named
  usercopy helpers
- nested trap state is architectural, not kernel-managed: `faultPC`,
  `previousMode`, `savedSUA`, `faultAddr`, `faultCause` are saved and
  restored by a per-CPU trap-frame stack, so kernel handlers do not need
  manual MFCR/MTCR save/restore dances to survive nested synchronous
  traps
- per-task quotas cap blast radius for pages, ports, waiters, shared
  mappings, and grants
- task exit sweeps all IPC structures that could otherwise leak state
  into a reused task id
- freed private and shared pages are scrubbed before reuse so no
  prior-task bytes reach the next owner
- shared mappings carry explicit read/write permissions, consumers map
  RO by default, and missing-permission `MapShared` is a hard error
- execute-only user pages are a real contract backed by loader tests
- host-root confinement in `bootstrap_hostfs.go` is complete against
  per-component symlink escape and case-normalization drift
- diagnostic landmarks (MMIO PTE enforcement, `CR_FAULT_PC` contract,
  kernel stack canaries on a minimal footprint, scoped atomic RMW
  plumbing) are test-pinned rather than tribal

M15.6 is intentionally **not**:

- not ASLR / KASLR / `ET_DYN` loading — post-M16
- not tagged TLB / ASID — performance, not hardening
- not shadow stack / CET-equivalent, PAC/BTI, MPK/PKU, MTE, CFI — each
  needs ISA + compiler + toolchain work and cannot be a "turn it on" win
- not a full M17-style grant sweep — that is M17's job; G4 here proves
  "no stale ownership after task recycling," which is narrower
- not a kernel text hash watermark — detection-not-prevention with JIT
  false-positive risk
- not `CR_TP` TLS plumbing — no consumer yet; revive with the
  stack-cookie milestone

## Relationship to M15.5 and M16

- M15.5 landed the substrate M15.6 rests on: task-exit hook plumbing,
  fault diagnostics with access-type / privilege / PTE bits,
  host-backed boot harness, MMIO carve-out documentation, A/D bit
  hardware handling, MTCR privilege gating.
- M15.6 is a pure kernel-surface hardening milestone layered on top.
- M16 Phase 1 does not begin until M15.6 G1–G6 are green. The
  "strongly recommended" R-series items ship with M15.6 but are not
  M16 gate conditions.

## Key Changes

### 1. JIT host-side W^X (G1)

`jit_mmap.go` currently maps the JIT region
`PROT_READ|PROT_WRITE|PROT_EXEC` permanently; `PatchRel32At` mutates
already-emitted code at runtime. The kernel's W^X story is contradicted
one syscall down.

M15.6 must:

- map the JIT backing pages twice: a writable view (RW, no execute) for
  emit and patch paths, and an execution view (RX, no write) used by
  JIT dispatch
- ensure at no point any view holds both `PROT_WRITE` and `PROT_EXEC`
- flush icache on the execution view on emit/patch so ARM64's
  split-I/D coherency is honored across the alias pair

The simpler fallback if dual-mapping is blocked (e.g., hardened kernels
without `memfd_create` `MFD_EXEC`) is `mprotect(RX)` at emit boundaries
with a flip back to RW only inside patch calls. Dual-mapping is
preferred; the fallback is acceptable when the host denies the
hardening-friendly path.

### 2. SMEP + SMAP-equivalent guards with mandatory usercopy helpers (G2)

This is one workstream spanning CPU changes, kernel changes, and a
migration audit. The goal is: every kernel access to user memory is
auditable, goes through a named helper, and runs inside an explicit
`SUAEN` / `SUADIS` region.

**CPU pieces.**

- `CR_MMU_CTRL` bit 2 `SKEF`: supervisor fetch from `PTE_U==1` page
  raises `FAULT_SKEF = 9`.
- `CR_MMU_CTRL` bit 3 `SKAC`: supervisor read/write on `PTE_U==1` page
  raises `FAULT_SKAC = 10`, unless...
- `CR_MMU_CTRL` bit 4 `SUA` latch: set by `SUAEN`, cleared by
  `SUADIS`; auto-cleared on trap entry so a kernel handler cannot
  inherit an open supervisor-access window from the interrupted code
  path.
- New privileged opcodes `SUAEN` / `SUADIS` (single-cycle latch ops).
- SKEF/SKAC checks land in `translateAddr` after the existing
  user/supervisor check, keeping fault causes distinct in diagnostics.

**Kernel pieces.**

- Canonical helpers in `sdk/intuitionos/iexec/iexec.s`:
  `copy_from_user(dst, user_src, len)`,
  `copy_to_user(user_dst, src, len)`,
  `copy_cstring_from_user(dst, user_src, max_len)`. Each wraps its
  access in `SUAEN` / `SUADIS` and delegates permission validation to a
  single named checker that returns canonical PTE flags.
- Every `SYS_*` slot that copies user data migrates onto the helpers.
  No direct supervisor load/store against a user pointer survives
  outside the helpers.
- SKAC is enabled at boot so any missed migration faults cleanly
  instead of silently succeeding.

**Trap stack (Phase 2c-trap — architectural, not kernel-managed).**

Earlier drafts of this plan spoke of a "nested-trap prologue" the
kernel would run to MFCR `CR_FAULT_PC` / `CR_SAVED_SUA` into a
kernel-stack save slot. That discipline was institutionalized debt:
any handler path missing the restore silently corrupted outer state.

M15.6 closes this by changing the CPU trap model. CPU64 owns a
small, fixed-depth trap-frame stack. `trapEntry` pushes the current
active frame (`faultPC`, `previousMode`, `savedSUA`, `faultAddr`,
`faultCause`) before overwriting; `ERET` pops after consuming. The
active fields remain the canonical "top of stack" read/written
through the existing `CR_FAULT_PC` / `CR_FAULT_ADDR` /
`CR_FAULT_CAUSE` / `CR_PREV_MODE` / `CR_SAVED_SUA` interface, so
kernel code continues to work and any existing manual save/restore
becomes redundant but harmless. Stack overflow halts the CPU with a
diagnostic: runaway nested faults are always a kernel bug and must
be visible, not silently dropped.

**Migration audit.**

Every user-memory touch point in the kernel is re-audited twice:

1. against the `validate_user_*` list (the bulk-copy sites)
2. against the non-validated touch points (boundary reads,
   single-field fetches, re-validation races between helpers)

Each migration carries a failing test pinning the new fault behavior
and a resource-leak test covering the TOCTOU window.

### 3. Per-task resource quotas (G3)

One misbehaving module task must not be able to exhaust shared pools.
M16 multiplies module tasks, shared mappings, and grant rows, so the
quota story must be in place before M16 Phase 1 begins.

Per-task counters, each with an explicit accounting contract:

- `QUOTA_PAGES` — private backing pages owned by the task; charged on
  private page grant, returned on free. Shared-mapping pages go under
  `QUOTA_SHMEM`.
- `QUOTA_PORTS` — live ports owned by the task; returned on destroy
  (including task-exit sweep).
- `QUOTA_WAITERS` — live waiter nodes keyed to this task; returned on
  wake or sweep.
- `QUOTA_SHMEM` — bytes currently mapped into the task; returned on
  unmap.
- `QUOTA_GRANTS` — live grant rows where this task is the grantor.
  Grantor-charged only; a grantee holding a handle does not consume
  `QUOTA_GRANTS`.

Defaults tuned for the 32 MB address space; per-task overrides at
create time. Each counter is inspectable via an exec query for
diagnostics only (not a stable API).

### 4. Exit-time IPC / waiter cleanup proof (G4)

`kill_task_cleanup` already clears owned ports. M15.6 extends the
task-exit hook to sweep a bounded, named set of IPC structures:

1. ports owned by the task (confirm the inline+overflow walk is
   complete and leaves no dangling reference)
2. waiter nodes keyed to the task (registry waiters, port
   receive-wait queues, wait-for-signal lists)
3. queued-message reply-port state whose reply port belongs to the
   exiting task — receiver's later `ReplyMsg` becomes a no-op drop,
   not a fault
4. M16 registry `waiters_head` rows — dead-waiter tolerance the M16
   registry relies on
5. grant rows keyed to the task: grants where the task is the
   grantor are revoked (row removed, backing reclaimed,
   `QUOTA_GRANTS` decremented) AND any `SYS_MAP_IO` PTEs the grantee
   already installed under that grant are torn down — in both the
   inline region rows and the per-task overflow chain — otherwise
   the grantee keeps live hardware access after the broker exits,
   defeating the revocation. Before each teardown the helper
   re-checks the grant chain: if another live grant still covers the
   same PPN range for the same grantee (e.g. a `GRANT_GRANTOR_NONE`
   bootstrap row alongside a broker-issued row), the mapping stays
   intact because it remains authorized. Grants where the task is a
   grantee have that task's handle/reference removed without touching
   any quota counter

Anything outside this list is explicitly M17 scope. A
build-tag-gated audit pass (`iexec_audit`) walks exactly these five
structures after task exit and asserts the exited `task_id` appears
nowhere.

### 5. Zero-on-free for private and shared pages (G5)

`FreeMem` and shared-mapping last-reference release must scrub the
backing page before releasing it. `MEMF_CLEAR` only guarantees
allocation-time zeroing; a freed page still carries prior-owner bytes.
Once M16 multiplies module churn, freed pages would routinely carry
prior-task data into the next owner — a real cross-task
confidentiality gap.

Tests target the security property directly ("a new owner cannot
observe prior contents"), not allocator placement. The scrub lives
in the page-release primitive so every call path inherits the
behavior.

### 6. Permission-preserving shared mappings (G6)

`MapShared` currently maps `PTE_P | PTE_R | PTE_W | PTE_U`
unconditionally. M16 has many buffers that should be read-only in
consumers (message payloads, descriptor copies). `MapShared` takes an
explicit permission bitmask (`MAPF_READ`, `MAPF_WRITE`; combinations
allowed). Producer maps RW on its side, consumers map RO. `PTE_W` is
set only when `MAPF_WRITE` is present. `PTE_X` is never set.

This is a deliberate IntuitionOS syscall ABI change, not a
compatibility shim. No "missing-mask means old default" fallback —
calls without a mask are a hard error. All in-tree callers are
migrated in this same milestone. Out-of-tree callers (if any) get a
migration note in the `IntuitionOS_Roadmap.md` M15.6 entry.

### 7. Strongly recommended, ship with M15.6 after the hard gate (R1–R9)

These items are M15.6 scope but do not gate M16:

- **R1** stack guard pages (user + kernel): one non-present page below
  each stack floor so overflow is a clean `FAULT_NOT_PRESENT`, not
  silent adjacent-page corruption.
- **R2** heap guard pages (opt-in via `MEMF_GUARD`): flanking
  non-present pages around allocations that opt in; M16 registry rows
  and port tables are natural first callers.
- **R3** integer-overflow audit on size arithmetic: `AllocMem`,
  `MapShared`, `CreatePort`, waiter-list insert, every `size + offset`
  / `count * stride` site in a 25-bit address space. Saturating
  checks.
- **R4** execute-only page semantics as a real contract:
  `validate_user_exec_range` today requires both `PTE_R` and `PTE_X`;
  update to require only `PTE_X`. Loader-level test maps a segment
  execute-only, executes from it, fails reads.
- **R5** HostFS per-component `NOFOLLOW` and case normalization;
  crafted symlink in a middle component is rejected; case variants
  resolve to one canonical host path.
- **R6** MMIO PTE enforcement test: walk every carve-out, assert
  each PTE has `PTE_U==0` and `PTE_X==0`. Makes M15.5's
  documentation into executable truth.
- **R7** `CR_FAULT_PC` contract + regression test: user-mode
  `MTCR CR_FAULT_PC` faults with `FAULT_PRIV`; document the
  "skip faulting instruction" semantics that justify kernel writability.
- **R8** kernel stack canary (demoted): ship a minimal version
  (~20 LOC) behind a build flag; do not present it as meaningful
  hardening beyond "catches one class". R1 already closes the main
  gap.
- **R9** scoped atomic RMW adoption (demoted): strictly scoped to
  the M16 module-registry row transitions and port / waiter head
  updates M16 Phase 1 introduces. Presented as "M16 prerequisite
  plumbing," not general hardening.

## Full-TDD Phase Plan

Phase ordering / dependency rule:

- Phase 0 locks scope and writes the M15.6 plan file (this document)
  and the roadmap entry.
- G1 is first because it rewrites the JIT memory backend; every later
  phase's tests can rely on W^X on the host.
- G2 sits next because the kernel-side migration and SKAC enable have
  the widest blast radius. The trap stack (Phase 2c-trap) ships as
  part of G2.
- G3 through G6 and all R-series items may overlap, subject to each
  item's failing-tests-first commit preceding its implementation
  commit.
- Phase N+1 closes the milestone only after all G items are green,
  the R items that landed are green, and every milestone-output doc
  has been updated and reviewed.

Each phase must show in git history:

1. a failing-tests commit (or equivalent demonstrable failing state)
2. the implementation commit
3. regression-verify: all earlier guards (M13, M14.x, M15.0–M15.5)
   stay green

### Phase G1: JIT host-side W^X

Write failing tests first for:

- after `ExecMem.Write` returns, the writable view has
  `PROT_READ|PROT_WRITE` and not `PROT_EXEC`
- the execution view has `PROT_READ|PROT_EXEC` and not `PROT_WRITE`
- dispatch runs against the execution view address, not the writable
  view address
- `PatchRel32At` targeting the writable view succeeds while the same
  address through the execution view is read-only

Implementation:

- prefer dual-mapping via `memfd_create` (with `MFD_EXEC|MFD_CLOEXEC`
  where available, falling back to plain `MFD_CLOEXEC` on hardened
  kernels that reject `MFD_EXEC`)
- split icache flush so `DC CVAU` targets the writable VA and
  `IC IVAU` targets the execution VA on ARM64; do not assume a single
  VA reaches both views
- provide an `mprotect(RX)` fallback path for platforms that cannot
  dual-map

Green:

- existing JIT regression suites (`TestAMD64_*`, `TestARM64_*`,
  `TestJIT_*`) stay green
- the new W^X test proves both views' permission invariant

### Phase G2: SKEF + SKAC + SUA with usercopy helpers

Write failing tests first for:

- SKEF fires on supervisor fetch from user page
- SKAC fires on supervisor read/write from user page with latch off
- same access inside `SUAEN` / `SUADIS` succeeds
- `SUAEN` / `SUADIS` silently-compiled-as-NOP regression guard in the
  JIT (`isBlockTerminator` + `needsFallback` + `emitBailToInterpreter`
  all fire for these opcodes)
- `CR_MMU_CTRL` bit mutation contract: MTCR ignores attempts to set
  or clear SUA; only `SUAEN` / `SUADIS` mutate it
- nested trap preserves outer SUA without kernel cooperation
  (architectural trap stack)
- nested trap preserves outer `FAULT_PC` / `PREV_MODE` / `FAULT_ADDR`
  / `FAULT_CAUSE` without kernel cooperation
- trap stack overflow halts cleanly
- static audit test: every `SYS_*` path that copies user data goes
  through the helpers
- per-syscall behavior regressions for every migrated slot
- TOCTOU regression: a shared-writable user page flipped between
  check and re-check surfaces as a clean fault, not silent
  corruption

Implementation:

- add `OP_SUAEN`, `OP_SUADIS` privileged opcodes
- add `SKEF`, `SKAC`, `SUA` bits to `CR_MMU_CTRL`
- add `FAULT_SKEF`, `FAULT_SKAC` fault codes
- extend `translateAddr` with SKEF on execute and SKAC on read/write
  paths, gated on the `sua` latch
- add the trap-frame stack and modify trap entry / `ERET` to
  push/pop the outer frame
- add `CR_SAVED_SUA` so monitor and existing kernel code that
  observes the latch snapshot continues to work
- migrate every `validate_user_*` site onto
  `copy_from_user` / `copy_to_user` / `copy_cstring_from_user`
- enable SKAC at boot

Green:

- every failing test above passes
- no kernel path outside the helpers touches a user pointer in
  supervisor mode

## Non-Goals

- ASLR / KASLR / `ET_DYN` loader — post-M16
- tagged TLB / ASID — performance optimization, not hardening; ASID
  layout would force a PTE break
- shadow stack / CET-equivalent, PAC/BTI, MPK/PKU, MTE, CFI — each
  needs ISA + compiler + toolchain work, not turn-it-on wins
- kernel text hash watermark — detection-not-prevention with JIT
  false-positive risk, disproportionate runtime cost
- `CR_TP` TLS plumbing — no consumer yet
- full M17-style grant sweep — M17's job; G4 here proves only "no
  stale ownership after task recycling"

## Assumptions and Defaults

- full TDD is mandatory; no item ships without failing tests first,
  implementation second, regression-verify third
- docs are required milestone outputs; M15.6 is not complete until
  every doc named above reflects the shipped change
- M15.4 and M15.5 are complete before M15.6 begins
- M15.6 keeps the strict-ELF `ET_EXEC` runtime contract
- M15.6 does not implement ASLR, KASLR, randomized placement, or
  `ET_DYN` loading
- M15.6 is the last kernel-hardening milestone before M16 Phase 1
  begins; after M15.6 lands, M16 may open its registry and
  lifecycle surface on a kernel that already handles the SMEP/SMAP,
  W^X, quota, and cleanup classes of mistake
