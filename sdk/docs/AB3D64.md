# AB3D64 internals

Engineer-facing companion to `sdk/ab3d64/README.md`. The README is
*how to use*; this doc is *why it looks the way it does*. Mirrors
`sdk/docs/m68Kto64.md`'s placement convention (port internals live
under `sdk/docs/`, not `sdk/ab3d64/`).

This is a living document. Sections lacking content are explicit
"not yet" markers — populate as the port reaches each milestone.

---

## 1. Scope and lineage

- **What the port is.** IE64 source-port of Alien Breed 3D II
  **Redux High Overdrive**, transpiled from upstream M68K via
  `m68kto64` and assembled with `ie64asm` into a single flat `.ie64`
  blob loaded at `PROG_START = 0x1000`.
- **Upstream commit pinned in snapshot.** `27f5cd4c2ddd` (alienbreed3d2 repo).
  Re-syncs MUST update this line and re-run `tools/amiga_strip.py`.
- **No Amiga surface.** Amiga library calls (`OpenLibrary`, `_LVO*`,
  `ExecBase`/`GfxBase`/`DosBase`/`IntuitionBase`, etc.) and NDK
  header includes (`exec/`, `dos/`, `graphics/`, `intuition/`,
  `devices/`, `hardware/`, `libraries/`, `resources/`) are stripped
  from the snapshot before transpile. If a future re-sync re-
  introduces any of them outside an `IFND IS_IE` gate, the build
  fails and the fix belongs in `tools/amiga_strip.py` glob list, not
  in a vendored NDK stub.
- **Out of scope.** Default (non-overdrive), `redux-low`, and SID
  music variants are not built in v1. See §7 for the restoration
  recipe.

## 2. Build pipeline

```
sdk/ab3d64/src/ie/ie_hires_platform.s ──┐
                                        ├──► m68kto64 (per-file)
sdk/ab3d64/src/ie/hires.s ──────────────┘            │
                                                     ▼
                                              concat (kmake.sh banner)
                                                     │
                                                     ▼
                                              ie64asm -o <out>.ie64
                                                     │
                                                     ▼
                            sdk/ab3d64/bin/ab3d2_ie64_redux_high_overdrive.ie64
                                                     │
                                                     ▼
                       IntuitionEngine -ie64 <path>  (loads at 0x1000)
```

- `m68kto64` is invoked once per input file with an independent
  symtab; results are concatenated with banner comments so ie64asm
  errors point back to the offending source file.
- `kmake.sh` (installed as `sdk/bin/m68kto64-kmake`) forwards `-I` to
  both tools and `-D` to `m68kto64`. Additional transpiler flags
  ride in via `M68KTO64_EXTRA_FLAGS` (this build uses
  `-strip-cond`, `-flag-liveness`, and the AB3D64 MMIO bridge range
  `-mmio-range 0xF0000-0xF2FFF`).
- Media path strings are emitted by `tools/run_media_prep.sh` into
  `build/redux-high/media_profile.i`, consumed by
  `ie/ie_file_io_runtime.i:997`.

## 3. Strip-list final form

`tools/amiga_strip.py` walks `src/` and applies two transforms:

1. **IS_IE-keyed conditional resolution.** For top-level
   conditionals whose predicate is `IS_IE`:
   - `IFD IS_IE` / `IFNE IS_IE,0` → keep body, drop wrapper.
   - `IFND IS_IE` / `IFEQ IS_IE,0` → drop body (or keep the `ELSE`
     branch if present).
   Other conditionals (e.g. `IFD IE_OVERDRIVE`, `IFNE CD32VER`) are
   preserved verbatim — the transpiler's preprocessor handles them
   at build time via `-D` defines.
2. **Amiga-include drop.** `include` directives whose path starts
   with any of `exec/`, `dos/`, `graphics/`, `intuition/`,
   `devices/`, `hardware/`, `libraries/`, `resources/` are deleted.

Initial run summary on the v1 snapshot: 13 files changed, 34 Amiga
includes dropped, 83 IS_IE-keyed regions resolved.

If a re-sync introduces a new Amiga symbol that survives the strip,
extend the include-prefix tuple in `tools/amiga_strip.py` and re-run.
Never vendor an NDK stub.

## 4. M68K → IE64 mapping notes specific to AB3D2

To be filled in as the runtime port progresses past assembly. Initial
data points:

- **PROG_START.** `0x1000`, matching upstream `vlink -Ttext 0x1000`
  and IE64 `machine_bus.go`.
- **Self-modifying code: none observed.** Plan §risk-2 mandates a
  spot-check for SMC (which `m68kto64` does not support). Audit
  performed on the pinned snapshot:
  - `modules/dev_inst.s` — does not exist in the snapshot.
  - `MakeFunc*` macros — no definitions or uses.
  - `move.b dN,LABEL+N` writes — all target data tables
    (`Prefsfile`, `LEFTPLAYEDTAB`, `RIGHTPLAYEDTAB`), not opcode
    bytes. Safe.
- **Two-TU concatenation.** `ie_hires_platform.s` provides ~100
  `xdef`s that `hires.s` `xref`s against. `m68kto64` drops xdef/xref;
  resolution happens at ie64asm time after concat. The top-level
  label intersection between the two TUs is empty as of the pinned
  snapshot — `make -C sdk/ab3d64 drift` will surface any new
  collisions on re-sync.
- **Register conventions, FPU usage, runtime memory map.** Not yet
  audited at runtime; the port has reached transpile but boot has
  not been characterised.

## 5. BSS strategy

`ie64asm` has no BSS section — `assembleDS` zero-fills the program
slice for `ds.b/.w/.l/.q` and advances `codeOffset`. Upstream
`vlink -b rawbin1` does the same for the `.ie68` flat blob, so the
apples-to-apples baseline is the upstream `.ie68` size.

**Measured on the pinned snapshot (2026-05-11):**

| Binary | Size |
|---|---|
| `ab3d2_source/ie/bin/ab3d2_ie68_redux_high_overdrive.ie68` (upstream) | 756 024 bytes |
| `sdk/ab3d64/bin/ab3d2_ie64_redux_high_overdrive.ie64` (this port)    | 2 228 828 bytes |
| Ratio | 2.95× |

Ratio is materially above the plan's 1.5× sanity threshold, so the
delta is **instruction-encoding bloat (m68k CISC → IE64 RISC)** rather
than BSS inflation: each m68k instruction word lowers to multiple
IE64 32-bit instruction slots (operand splits, shadow-CCR
maintenance, address-mode normalisation).

**Decision for v1: accept and defer.** Options if this proves
unacceptable downstream:

- (a) Extend `ie64asm` with a real out-of-band BSS section emitted as
  metadata and zero-allocated at load.
- (b) Relocate the large textured-buffer / sound-mixer reservations to
  a runtime `malloc`-equivalent in the IE64 ABI.
- (c) Tune the transpiler's lowering density (e.g. fuse shadow-CCR
  bookkeeping across more spans).

(a) is the cleanest if file size becomes a distribution gate. (b) is
the cleanest if it's runtime memory pressure. (c) is generic and
benefits every port, but requires deeper transpiler work.

### ie64asm side-fixes that landed for this port

- `assembler/ie64asm.go` — `__m68kto64_*` continuation labels now skip
  `lastGlobalLabel` updates in both pass-1 and pass-2 label handling
  blocks so neighbouring `.local:` labels stay in scope.
- `parseDispReg` walks the **last** paren-pair when picking the
  register slot, so disp expressions of the form `(A*B)+C(r11)`
  no longer mis-parse the inner expression as a register.
- Binary-literal lexer accepts the m68k `%1010_…` prefix in
  expressions (sister of `$hex` and `0x…`).
- Character/string literal lexer accepts both `'…'` and `"…"` and
  packs up to 8 bytes BE into the immediate, matching m68k's
  `move.l #'.pal',d0` idiom. `dc.b` lays string contents down as
  successive bytes rather than packing.

### Transpiler-side preprocessor fix that landed for this port

- A label preceding a macro invocation (`LABEL: MACRO args`) is now
  emitted by the preprocessor before the expanded body, so labels
  attached to macro call-sites survive expansion (previously dropped
  silently). See `cmd/m68kto64/preproc.go` macro-dispatch path.

These belong to the IE64 assembler / transpiler proper, not the port,
and are documented here because AB3D64 is what motivated them.

### Verification status

`make ab3d64` (or `make -C sdk/ab3d64 all`) successfully transpiles +
assembles `bin/ab3d2_ie64_redux_high_overdrive.ie64` on the pinned
snapshot. `bash sdk/ab3d64/tools/run_boot_audit.sh /tmp/ab3d64_audit.log`
now reaches `_ie64_entry`, init, palette/HUD setup, `Game_Begin`,
`DrawDisplay`, and `_Vid_Present`, then reports
`[ASRT] fb nonzero pixels = 4096 (want > 0)` and
`[DONE] all landmarks fired with assertions green` on the headless VM.
Pixel/hash parity and sustained full-render frame rate remain outside this
milestone.

Asset staging caveat: the `_build/ie_menu/`, `_build/ie_unpacked/`,
and `_build/ie_media/` trees were copied verbatim from upstream's

## 7. Restoring non-overdrive / redux-low / SID variants

The Phase 0.5 strip leaves all upstream variant gates intact (only
`IS_IE` is resolved). No source changes needed; only Makefile changes:

1. Add a per-variant `DEFINES_*` block, e.g.
   `DEFINES_REDUXLOW := -D mnu_nocode=1 -D USE_16X16_TEXEL_MULS=1`
   (omit `-D IE_OVERDRIVE` for non-overdrive; add `-D IE_ENABLE_SID_MUSIC=1`
   for SID).
2. Add a target per variant that invokes
   `tools/run_media_prep.sh redux-low build/redux-low` and the
   appropriate `-I build/<profile>` plus DEFINES.
3. The SID variant additionally requires the SID assets to be
   present under `media/redux-{high,low}/` (out of scope for v1
   build infrastructure).

## 8. Porting another AB3D2-style M68K codebase

Distilled checklist from this port:

1. **Run `m68kto64`'s corpus integration test first**
   (`go test -tags m68kto64_integration -run TestRealworldCorpus`).
   The failure list — under default seeds with no `-D` — is the real
   shape of the work, not what speculation suggests.
2. **Audit Amiga surface before writing a transpiler patch.**
   Grep for `_LVO*`, library bases, OpenLibrary callsites, and Amiga
   NDK includes. Every hit must be inside an `IFND IS_IE` branch (or
   equivalent); if not, that's a real port gap.
3. **Write a strip pass before reaching for the transpiler.** Most
   first-pass failures are dead Amiga code that should never have
   reached `m68kto64`. The strip is cheaper and more legible than a
   transpiler stub for unused symbols.
4. **Land transpiler features upstream, never as port-local patches.**
   `cmd/m68kto64/` is the contract; every category-2 fix needs a
   unit test there and a doc update in `sdk/docs/m68Kto64.md` in
   the same commit. Out-of-sync doc is a bug.
5. **`-strip-cond` is your friend.** Inactive conditional branches
   produce noisy unknown-mnemonic errors that obscure real gaps.
   Use `-strip-cond` for the build; keep wrappers only when debugging
   the strip-list.
